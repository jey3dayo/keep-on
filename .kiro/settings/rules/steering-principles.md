# Steering Principles

## 目的

- Steering は「意思決定を助けるパターンと原則」を残すための知識ベース
- 網羅的な一覧や細かな実装メモは避け、再利用性の高い判断材料に絞る

## 記述の粒度

- 新しいコードが既存パターンに従う限り、Steering を更新不要とする
- 具体例は最小限（1〜2例）に留め、ルールの意図を説明する
- 「Bad: すべてのファイル列挙 / Good: 代表例と構成パターン」を守る

## 更新方針

- 既存のユーザー記述は尊重し、基本は「追記」で対応（置換は慎重に）
- 破壊的変更や大規模な方向転換は、必ず提案→確認の順で進める

## ドリフト検知

- Steering とコードの差分は「警告」「更新候補」「推奨」に分けて報告
- 軽微な差分（命名や細部の変更）は原則スルー

## セキュリティ

- API キー、パスワード、トークン、接続文字列などの機密は記載しない
- 内部運用の秘匿情報は Steering に含めない

## 対象外の情報

- `.kiro/settings/` はメタデータのため Steering に記載しない
- エージェント固有のツール/フォルダ（例: `.cursor/`, `.gemini/`, `.claude/`）は記載しない

## 例外時の扱い

- 不確実な場合は「仮説」と明記し、確認質問を行う
- 情報不足なら無理に補完せず、必要な確認点を短く提示する
